SITE AGENT - OFFLINE BEHAVIOUR:

    LOCAL STATE FILE (encrypted JSON):
        {
            "last_sync": "2024-01-01T12:00:00",
            "approved_accounts": [...],     # last known good config
            "temp_access": [...],           # active temp passwords
            "cameras": [...],               # local camera list
            "pending_sync": [...]           # actions to sync when back online
        }

    ONLINE:
        - Pull config delta from central on startup
        - Pull config delta every 60 mins
        - Push status heartbeat every 60 mins
        - Push audit logs every 60 mins (batched)
        - Receive temp password requests in real time

    OFFLINE:
        - Continue enforcing approved_accounts from last known config
        - Continue expiry thread, reset passwords as they expire
        - Queue audit logs locally
        - Queue status updates locally
        - When back online, flush queue to central

    COMMS PATTERN:
        - Site agent always initiates connection (no inbound ports needed)
        - Central API is the only thing that needs to be publicly reachable
        - Site agent polls for pending commands (temp password requests)
            GET /api/v1/sites/{site_id}/pending
        - Reduces need for central to push to sites


TEMP PASSWORD FLOW:

    CENTRAL:
        1. User requests temp password via web app
        2. Backend generates secure password (secrets module)
           - 16 chars minimum
           - Upper, lower, numbers, symbols
        3. Backend stores in central DB:
            {
                "request_id": "uuid",
                "site_id": "site-001",
                "username": encrypted(username),
                "password": encrypted(password),  # for re-push if needed
                "expiry": "2024-01-06T14:32:00",
                "requested_by": "user@company.com",
                "status": "pending"
            }
        4. Password returned to web app, displayed once, never again
        5. Request marked as "delivered"

    SITE AGENT:
        1. Polls central for pending commands
        2. Receives temp password request
        3. Decrypts password
        4. Sets password on all local cameras via ONVIF
        5. Stores in local state (encrypted)
        6. Confirms back to central (status: "applied")

    EXPIRY:
        1. Expiry thread runs every 60 mins
        2. Checks local state for expired temp passwords
        3. Generates new random password (not stored, not returned)
        4. Sets new password on all cameras
        5. Removes from local state
        6. Logs expiry event
        7. Notifies central (status: "expired")
